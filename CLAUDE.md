When running Python directly from the command line (e.g. `python3 -c "..."` or `python3 script.py`), use `uv run python3` instead. This does NOT apply to `python3` invocations inside bash scripts — those work as-is.

## Workflow — making changes locally

1. Edit the `my-*.txt` files (and docs/scripts if needed)
2. Run `bash apply-exclusions.sh` to verify — Stage 7 auto-restores blocklist files
3. `git pull --rebase` before committing — the GitHub Actions weekly sync may have pushed new commits to `origin/master` while you were working
4. Commit only the files listed below
5. Push

## Commit rules — what to commit and what NOT to commit

Only the following files should be committed manually:

- `my-exclusions.txt`, `my-inclusions.txt`, `my-additions.txt` — the three config files
- `apply-exclusions.sh` — the script itself (if modified)
- `CONTEXT.md`, `CLAUDE.md`, `README.md` — documentation (if modified)
- `.github/workflows/sync-upstream.yml` — the workflow (if modified)
- `.claude/` — Claude Code settings

NEVER commit blocklist output files manually. These are generated by `apply-exclusions.sh` and should only be committed by the GitHub Actions workflow during upstream sync:

- `blocklist.txt`, `domains.txt`, `wildcard-blocklist.txt`
- `unbound-blocklist.txt`, `rpz-blocklist.txt`
- `pihole-blocklist.txt`, `hosts-blocklist.txt`
- `little-snitch-blocklist.lsrules`
- `install.html`, `little-snitch-install.html`

Running `apply-exclusions.sh` locally is safe — it automatically detects local runs, verifies the changes, and restores all blocklist files to their git state. You will never have dirty blocklist files after a local run.
